Abstract

The Smart Travel Planner Application is a complete travel-management system developed using Streamlit as the presentation layer and SQLite as the persistent data store. The project was implemented as a production-style academic application rather than a prototype. It includes user registration, secure login, preference capture, trip creation, itinerary generation, budget estimation, destination recommendations, profile management, analytics, administrative controls, exports, and centralized logging.

The system is designed to solve a common planning problem: travelers often need to compare destinations, estimate costs, organize schedules, and retain a historical view of their trips. The application addresses that problem through a unified workflow in which the user can define preferences, evaluate recommendations, generate a day-wise trip plan, and monitor spending assumptions through an integrated dashboard.

This report studies both the product and the implementation. It describes how the codebase is organized, how tables are related in the database, how security is implemented for credentials, and how each page uses shared service modules to avoid duplication. The result is a coherent reference that can support academic review, viva preparation, project demonstration, and future maintenance.

Background and Motivation

Travel planning normally involves scattered tools: spreadsheets for budgets, maps for places, notes for plans, and separate portals for bookings. That fragmentation creates practical issues. Users lose historical context, cannot compare destinations consistently, and often build itineraries manually with weak cost discipline.

The motivation behind Smart Travel Planner is to centralize these concerns into one lightweight but structured application. The design goal is not to replace airline or hotel booking platforms. Instead, it acts as an intelligent planning and management layer before or alongside bookings. The application helps users answer questions such as: Which destination fits my budget and interests? What might a five-day plan look like? How much should I expect to spend? Which trips did I plan previously and what were their financial outcomes?

From an academic viewpoint, this project demonstrates the integration of database design, security, recommendation logic, analytics, structured logging, modular programming, and user-facing workflows in a single system.

Problem Statement

The core problem addressed by the project is the absence of a simple planning platform that combines destination selection, budget forecasting, itinerary preparation, and historical trip management. Existing consumer tools typically focus on booking or travel discovery rather than integrated planning. They also do not usually expose project-friendly, inspectable logic suitable for educational review.

This system therefore targets a defined problem statement: design and implement a travel planning application where authenticated users can manage personalized travel preferences, create detailed trips, generate estimates and schedules automatically, and access analytics over time, while an administrator can supervise system activity.

The problem is handled through modular services rather than monolithic page logic. This decision matters because it keeps the user interface thin and allows the core planning logic to be tested independently of the web pages.

Architectural Style

The application follows a pragmatic layered structure. Streamlit page files are responsible for rendering forms, tables, charts, and user messages. They do not directly implement business logic. Instead, they call into service modules located in the smart_travel package.

Those service modules coordinate validation, data access, algorithmic processing, and audit recording. The database layer is abstracted into helper functions for query execution and transaction handling. This pattern gives the project several benefits: cleaner UI code, reusability, easier testing, and lower risk of accidental duplication between pages.

Page 14 of 80

High-Level Flow

At startup, the main application imports initialize_app() from the package. Initialization creates the log directory if necessary, configures the log handler, initializes the SQLite schema, and ensures the existence of the default administrator account. After that, Streamlit session state is prepared and the page UI is rendered.

During normal use, the flow is: authenticate user, read profile and preferences, generate recommendations, create or edit trips, calculate budgets, generate itinerary rows, write all related records in the database, and display them back to the user through the dashboard, history, and analytics pages.

Users Table Design

The users table stores the primary identity record. It contains a generated integer primary key, the user's full name, unique email address, password hash, salt, role flag, active flag, home city, preferred currency, created timestamp, and last login timestamp.

Storing the password hash and salt separately is the correct security approach for this scale of application. The table also includes is_admin and is_active booleans, which are used by the admin panel and authentication logic to govern system access.

Page 29 of 80

User Preferences Table Design

The user_preferences table is a one-to-one extension of the user record. It stores the travel style, budget level, a JSON array of interests, and the preferred continent. This is kept separate from the identity table because preferences change independently from authentication data.

Using a dedicated table also makes the design more normalized and avoids mixing security-sensitive columns with planning preferences in the same structure.

Trips Table Design

The trips table is the central domain table. It stores the owning user ID, title, destination, destination country, origin city, start and end dates, traveler count, budget limit, selected interests as JSON, trip type, travel style, notes, status, estimated cost, actual cost, and budget breakdown JSON.

This table intentionally stores a financial snapshot rather than recalculating historical budgets every time a user opens a trip. That preserves the assumptions made when the trip was created or last edited.

Trip Days and Recommendation Tables

The trip_days table stores day-level itinerary output. Each record has the trip ID, day number, date, day title, morning plan, afternoon plan, evening plan, and estimated daily cost. This isolates repeating schedule content from the trip master row.

The trip_recommendations table stores ranked alternatives generated at save time. Keeping them persistently allows a user to revisit the context in which a trip was planned rather than losing the recommendation set.

Password Security Design

Password storage uses hashlib.pbkdf2_hmac() with the SHA-256 digest and 200,000 iterations. A unique random salt is generated for each password unless an existing salt is being reused for verification.

The resulting hash and salt are stored as hexadecimal strings in the database. Password comparisons use hmac.compare_digest(), which is the correct approach for reducing timing-attack leakage in direct string comparisons. For a local academic project, this is a solid and defensible implementation.

Page 36 of 80

Registration Workflow Internals

The register_user() function performs several steps in sequence. It validates the input payload, checks whether the email already exists, hashes the password, inserts the user record, inserts the preference record, writes an informational log entry, and records a registration audit event.

If validation fails, the function returns a user-facing message and also records the failure. If insertion fails, the function logs the exception and returns a safe error message instructing the operator to consult the log.

Page 37 of 80

Login Workflow Internals

The login flow first logs the attempt, retrieves the user row by normalized email, checks whether the user is active, derives the expected hash from the supplied password and stored salt, and compares it to the stored hash. On success, the function updates last_login_at, records an audit event, and returns the user object.

On failure, the code distinguishes between unknown email, inactive user, and bad password. That distinction is useful in logs and audit data even though the user-facing message remains appropriately generic.

Logout and Session Flow

Session handling is managed in smart_travel/ui.py. The helper init_session_state() sets predictable defaults, while set_session_user() writes authenticated identity details into st.session_state. The clear_session_user() helper resets those values and records logout.

This is a sensible session model for a Streamlit application because Streamlit provides server-managed state per session. Access protection is then enforced through require_login(), with an optional administrator-only branch.

Budget Engine Overview

The budget engine resides in smart_travel/budget.py. Its primary public function is estimate_trip_budget(). This function receives destination attributes, dates, traveler count, travel style, trip type, and budget limit, and returns a structured dictionary containing multiple cost categories and whether the estimate falls within the declared budget.

The engine is intentionally transparent. Instead of using opaque heuristics, it combines destination-level cost fields with style multipliers, trip-type activity factors, a room factor, and a contingency allowance.

Page 40 of 80

Budget Formula Details

The budget model estimates these components:

  • Stay cost based on average daily destination cost, trip days, room factor, and style multiplier
  • Meal cost based on daily meal estimate, days, travelers, and style multiplier
  • Local transport cost based on daily estimate and traveler count
  • Activity cost based on daily activity estimate and trip-type factor
  • Flight cost using a destination-level base flight estimate
  • Visa cost using destination-level visa fee
  • Contingency at 8 percent of the subtotal

This structure is simple enough to explain yet rich enough to feel realistic. The returned breakdown is stored in JSON so it can be displayed later without recalculation.

Page 41 of 80

Itinerary Engine Overview

The itinerary logic is implemented in smart_travel/itinerary.py. Its role is to convert a destination, date range, and list of interests into a multi-day travel plan. The engine works by mapping interests to activity categories and then selecting matching activities from the destination dataset.

If the destination does not have activities matching the requested interests, the system falls back to the full destination activity pool. This fallback avoids empty itineraries and ensures every saved trip is complete.

Page 42 of 80

Itinerary Logic Details

Each itinerary day contains a title, date, morning segment, afternoon segment, evening segment, and estimated daily cost. The engine treats the first and last days slightly differently when the trip spans multiple days: the first day emphasizes arrival, and the last day emphasizes final experiences and departure.

Activities are selected cyclically from the filtered activity pool. This means the logic can support trips longer than the count of unique activities without failing. The daily cost is computed using activity costs plus meal and local transport estimates from the destination.

Page 43 of 80

Recommendation Engine Overview

The recommendation module in smart_travel/recommendation.py produces ranked destination candidates. It accepts budget, travelers, dates, interests, trip type, and an optional preferred continent. For each destination in the dataset, it estimates a budget and then computes a score based on budget fit, interest alignment, trip-type support, seasonal suitability, and continent preference.

The recommendations are returned as ranked dictionaries with score, summary, estimated total, and budget-fit flag.

Page 44 of 80

Recommendation Scoring Logic

The scoring model is rule-based rather than machine-learned. That is appropriate here because the data is a small curated dataset, not a large historical corpus. A destination receives score contributions for:

  • Being within the user's budget or near the budget threshold
  • Matching the user's preferred continent
  • Being seasonally aligned to the requested start month
  • Supporting the selected trip type
  • Carrying high category scores for the user's interests

The result is explainable and easy to maintain. Every output can be justified by visible scoring rules.

Service Layer Overview

The file smart_travel/services.py ties together the application. It exposes initialization, trip persistence, list and detail retrieval, deletion, status updates, administrator updates, log reading, and aggregate helper functions.

This module is the application's orchestration hub. It is where multiple lower-level services are combined into user-facing workflows. That design is important because it gives the project a true backend layer despite using Streamlit for the frontend.

Page 46 of 80

Trip Creation Transaction

The most important service function is save_trip(). For a new trip, it validates the payload, retrieves the destination, calculates the budget, builds the itinerary, generates alternative recommendations, inserts the trip master row, inserts trip-day rows, inserts stored recommendations, logs success, and writes an audit event.

This flow is intentionally transactional. The related records are written within a managed cursor block so that partial trip states are avoided.

Page 47 of 80

Trip Update Transaction

Trip updates reuse the same save_trip() function but with a trip ID. In that branch the service updates the master trip row, deletes existing itinerary and recommendation children, and regenerates them from the current payload. This keeps all derived trip artifacts synchronized with the latest user choices.

The design trades a small amount of rewrite cost for correctness and simplicity. Given the low scale of the application, this is the right choice.

Page 48 of 80

Trip Delete and Status Workflows

The delete path first checks whether the trip exists and whether the requesting user is permitted to remove it. If the call comes from an administrator, the ownership restriction can be bypassed explicitly through the admin_delete argument. Deletions are recorded in the audit log.

Financial and status updates are handled by update_trip_financials(), while administrative status updates use update_trip_status_as_admin(). This distinction preserves role semantics in the codebase.

SQL queries in the module calculate total trip counts, upcoming trip counts, completed trip counts, spend totals, monthly trends, spend by destination, and aggregate administrative metrics.

Performance Considerations

The application is lightweight, and its performance characteristics are favorable for the intended scale. SQLite is fast for small to medium local workloads. The destination dataset is cached in memory after first load. Queries are simple and indexed implicitly through primary keys and unique constraints.

The most expensive operations are itinerary regeneration and recommendation scoring, but even these are modest because the destination dataset is small and deterministic.

Page 70 of 80

Scalability Considerations

The current design is optimized for local or low-concurrency educational usage. If the system needed to scale, SQLite would be the first component to reconsider. A networked relational database such as PostgreSQL would be a more appropriate backend for concurrent multi-user traffic.

Even so, the service structure would support that migration reasonably well because database interactions are mostly concentrated in a dedicated module and in analytics queries.

Page 71 of 80

Maintainability Considerations

The codebase is maintainable because responsibilities are separated. Validation is not mixed with UI rendering, database statements are not scattered across every page, and business logic such as budgeting and itinerary generation resides in dedicated modules. Enumerated values are centralized in configuration.

This means future features can be added with predictable extension points rather than widespread edits.

Current Limitations

The project intentionally avoids certain advanced features. It does not integrate airline or hotel APIs, does not support collaborative trip editing, does not provide map rendering, and does not perform route optimization. The recommendation engine is rule-based and dataset-driven, not machine-learned.

These are limitations in product scope, not implementation defects. Within the defined scope, the project is complete.
